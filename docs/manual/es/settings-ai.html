Configuración → IA

Principal → Chat usa el proveedor seleccionado aquí. Las opciones locales primero (llama.cpp, Ollama) mantienen el diario y los mensajes en tu máquina. Los proveedores en la nube envían el chat a sus API — úsalos solo si aceptas esa compensación. Elige Ninguno para desactivar el chat hasta que estés listo.

IA local (llama.cpp)

Moni puede descargar un binario llama-server y un modelo GGUF, y gestionar el proceso del servidor en 127.0.0.1 (puertos 8081–8299). Flujo recomendado la primera vez: pulsa Set up local IA o descarga binario y modelo por separado, luego Iniciar servidor.

Controles clave

  • Modelo a usar — elige un ajuste predefinido u otro archivo GGUF. Cambiar el modelo con el servidor en marcha puede pedir reinicio.
  • Usar GPU si está disponible — requiere una compilación llama-server con GPU; elige un dispositivo GPU o «All GPUs».
  • Desactivar razonamiento del modelo — recomendado para chat; pasa --reasoning off para que las respuestas aparezcan en el panel (necesario para Qwen 3 y modelos «thinking» similares).
  • Cambiar modelo sin reiniciar (modo router) — cambia modelos sin reinicio completo con un diseño --models-dir multi-modelo.
  • Auto-start local IA on launch — inicia el servidor gestionado tras cargar la app (binario y modelo ya instalados).
  • Chat reply length / timeout — limita tokens y tiempo de espera en hardware lento.
  • Directorio del binario / Directorio de modelos — sobrescribe rutas por defecto; Restablecer rutas a valores predeterminados restaura el diseño incluido.
El servidor gestionado por Moni usa el primer puerto libre en localhost de 8081 a 8299. El puerto se guarda con la identidad del proceso para que Moni se reconecte tras reiniciar.

Ollama

Apunta Moni a tu instancia Ollama en ejecución (URL base por defecto en localhost). Selecciona un modelo de la lista que expone Ollama. Asegúrate de que Ollama esté instalado y el modelo descargado (ollama pull …) antes de chatear.

Proveedores en la nube (OpenIA, Claude, Gemini)

Introduce tu clave API (guardada en configuración local). Elige modelo y opciones del proveedor en el panel. El chat en la nube no requiere GPU local pero envía el contenido de la conversación a la API del proveedor — no lo uses para diarios que no puedas compartir externamente.

Ajustes hledger relacionados (Configuración → hledger): Verificar comandos hledger en respuestas de IA valida ejemplos del asistente contra la complist de tu versión y tu binario antes de pulsar Run.

IA Wrapper (experimental)

Se conecta a la aplicación de escritorio Runable IA Wrapper en la misma máquina. Registra un cliente en IA Wrapper → API, activa servicios en Preferences y pega el token de registro en Moni. Elige servicio, modo (webview Electron o extensión Chrome) y etiqueta de cliente. Pulsa Conectar para listar servicios disponibles.

Contacta con Runable.app para acceso a IA Wrapper si no tienes la aplicación complementaria instalada.

Documentación hledger (RAG)

Moni puede incrustar documentación hledger para que la IA local cite flags y conceptos correctos. Las líneas de estado en el panel IA muestran:

  • No disponible — opcional para IA local; la nube puede funcionar sin ello.
  • Estado: archivo en disco — índice presente, se carga al enviar el primer mensaje.
  • Listo — N fragmentos de documentos — la recuperación está activa para respuestas fundamentadas.

Para rutas, orden de búsqueda y detalles de complist, revisa Configuración → Registros → Depuración en app.log, o el botón 📜 Registros del proveedor en el panel IA.

Comportamiento del chat en Principal

Con un proveedor conectado, escribe en Principal → Chat. Moni puede sugerir comandos hledger ejecutables; pulsa Enter o usa los controles para ejecutarlos en el panel Libro mayor. Con autocompletado activado (Configuración → hledger), al escribir argv de hledger aparece un popover; Ctrl+H ejecuta hledger desde la línea de entrada.

El registro de interacción (Configuración → Registros) puede guardar entrada del usuario, conversación con IA y resultados del panel ledger para tu propia auditoría.

Resolución de problemas

  • La API compatible con OpenIA no es accesible. — para llama.cpp, confirma que Iniciar servidor tuvo éxito; abre el log del servidor desde el LED de diagnósticos. Comprueba que el firewall no bloquee localhost.
  • Wrong GPU build — un binario solo CPU no puede usar GPU; descarga o instala una variante llama-server con GPU.
  • Empty or “thinking” replies — activa Desactivar razonamiento del modelo y reinicia el servidor.
  • Cloud errors — verifica clave API, nombre del modelo y red; lee 📜 Registros para el estado HTTP.
  • RAG not loading — revisa líneas de depuración en app.log; el primer chat puede tardar más mientras se carga el índice.

Copia el diagnostics bundle de llama.cpp (estado + cola de log) al reportar problemas — incluye lo que soporte necesita sin exponer tu diario.
